# ADR-0001: 采用 Agent + Skill + Orchestrator 的 Conduit 交付架构

## 状态

已接受 (Accepted) — 2026-05-20

## 背景

课题要求在 Conduit 开源全栈仓库上实现一个“超级个体”，完成从 PM 自然语言需求到代码写入、自动化验证和 PR 提交准备的闭环。

核心挑战是：

1. PM 需求通常模糊，需要先澄清。
2. Conduit 是真实仓库，不能用 mock 仓库替代。
3. 生成代码必须可验证，不能只展示模型输出。
4. 新需求模式需要可扩展，不能每次改主流程。

## 决策

选择 **Orchestrator 串联阶段 + Agent 执行职责 + Skill 承载需求模式 + Sandbox Adapter 操作 Conduit**。

不选择：

- 只做聊天 UI：无法证明真实仓库写入和验证。
- 全自动无人值守：风险过高，不符合人工可介入要求。
- 为 P0 搭建复杂平台：会偏离 3 周挑战目标。

## 核心决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 主流程 | clarify → plan → edit → verify → pr | 对齐源题闭环，便于演示 |
| 需求模式 | Skill 注册 | 新模式优先新增 Skill，不改主干 |
| 仓库操作 | Conduit Sandbox Adapter | 集中封装 read / patch / diff / command |
| 人工介入 | 需求、方案、PR 前确认 | 控制高风险节点 |
| 验证方式 | 使用 Conduit 真实脚本 | 避免伪成功 |
| 证据 | run 维度归档，submission 维度汇总 | 便于答辩、审计和材料提交 |
| AI 留痕 | 记录模型、tokens、延迟、成本和人工审阅点 | 对齐官方 AI 使用要求和可观测性加分项 |

## 影响

- 实现仓库必须包含前端、后端、AI 编排三层。
- P0 至少实现一个 L1 Skill。
- 所有代码变更必须能用 git diff 展示。
- lint / test 缺失或失败必须显式显示。
- PR 草稿必须包含需求、方案、变更文件、验证结果和风险。
- 最终材料必须能汇总 Demo、视频、README、架构图、AI 使用说明和工程难点说明。

## 结论

该架构足够小，可以在 3 周内落地；同时保留 Skill 扩展、断点重放和跨栈一致性的 P1 演进空间。
