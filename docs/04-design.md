# 体验设计文档：Conduit 超级个体端到端交付系统 v1.0.0

## 设计目标

本系统的界面不是普通聊天窗口，而是 PM、Agent 流程和 Conduit 仓库变更之间的交付控制台。设计目标是让用户清楚看到：需求是否澄清、方案是否可执行、代码改了哪里、验证是否通过、是否可以提 PR。

## 核心原则

- **先澄清再生成**：模糊需求必须先显示问题和假设。
- **先方案再写入**：代码生成前展示模块定位与影响范围。
- **证据优先**：每个阶段结论都附带文件、diff、命令或日志。
- **人工可介入**：澄清、方案、代码、测试失败都能暂停并修改输入。
- **单屏可演示**：答辩时能用一屏讲清当前阶段、结果和下一步。

## 用户路径

```text
PM 输入需求
  ↓
澄清问题 / 默认假设
  ↓
确认需求卡片
  ↓
方案拆解 + 模块定位
  ↓
生成代码并写入 Conduit sandbox
  ↓
运行 lint / 单测
  ↓
展示 diff + 验证结果 + PR 草稿
```

## 页面 1：需求对话页

### 目标

承接 PM 自然语言输入，并展示澄清 Agent 的问题、假设和需求卡片。

### 布局

```text
┌────────────────────────────────────────────────────────────┐
│ Conduit Super Individual                                  │
├───────────────────────┬────────────────────────────────────┤
│ 阶段进度              │ 对话区                             │
│ 1 需求输入            │ PM: 给文章列表加阅读量展示          │
│ 2 澄清确认            │ Agent: 需要确认阅读量来源...        │
│ 3 方案拆解            │                                    │
│ 4 代码生成            │ [确认需求] [补充约束]               │
│ 5 验证提测            │                                    │
├───────────────────────┴────────────────────────────────────┤
│ 当前需求卡片：目标 / 范围 / 非目标 / 验收标准               │
└────────────────────────────────────────────────────────────┘
```

### 关键状态

| 状态 | 含义 | 用户动作 |
|------|------|----------|
| waiting_input | 等待 PM 输入 | 输入需求 |
| clarifying | Agent 正在澄清 | 等待或补充 |
| needs_confirm | 需要确认需求卡片 | 确认 / 修改 |
| rejected | 需求不可执行或信息不足 | 重新描述 |

## 页面 2：方案与模块定位页

### 目标

让用户在生成代码前确认系统理解了仓库结构和影响范围。

### 内容

- 需求摘要
- 匹配的 Skill
- 受影响模块
- 目标文件路径
- 风险与假设
- 将执行的验证命令

### 示例

```markdown
## 方案摘要

**需求**：文章列表卡片展示阅读量（§9.1：前端假数据，不改后端）。
**Skill**：article-list-display-field
**影响范围**：前端文章列表组件。
**目标文件**：
- `frontend/src/components/ArticlesPreview/ArticlesPreview.jsx`
- `frontend/src/styles.css`

**验证**：
- `npm run lint:sandbox`
- `npm test --prefix sandbox-repo`
```

## 页面 3：执行监控页

### 目标

展示 Orchestrator 当前阶段、Agent 输出、事件日志和可介入点。

### 阶段卡片

| 阶段 | 显示内容 |
|------|----------|
| clarify | 澄清问题、确认结果 |
| plan | 方案、模块定位、Skill |
| edit | 文件写入、diff 摘要 |
| verify | 命令、退出码、日志 |
| pr | PR 标题、描述、风险 |
| submit | Demo、视频、仓库、README、AI 使用说明、工程难点 |

### 失败呈现

失败必须显示：

- 失败阶段
- 失败命令或 Agent
- 原始错误摘要
- 已生成的中间工件
- 可选动作：修改需求、**从阶段重放**（`resume-from-stage`）、`retry` 新 run、停止流程

## 页面 4：Diff 与验证结果页

### 目标

让评审快速判断代码是否真实写入 Conduit，并是否具备提测条件。

### 布局

```text
┌────────────────────────────────────────────────────────────┐
│ 变更文件                                                   │
│ - frontend/src/components/ArticlesPreview/ArticlesPreview.jsx│
│ - frontend/src/styles.css                                  │
├────────────────────────────────────────────────────────────┤
│ Diff 摘要                                                  │
│ + 展示阅读量 icon 和数字                                   │
│ + 缺少 slug/title 时显式报错                               │
├────────────────────────────────────────────────────────────┤
│ 验证                                                       │
│ ✅ npm run lint:sandbox                                     │
│ ✅ npm test --prefix sandbox-repo                           │
└────────────────────────────────────────────────────────────┘
```

## 页面 5：PR 草稿页

### 目标

汇总可提交 PR 的材料，支持复制或一键提交。

### 字段

| 字段 | 说明 |
|------|------|
| branch | 建议分支名 |
| title | PR 标题 |
| summary | 变更摘要 |
| files_changed | 变更文件 |
| tests | 验证命令与结果 |
| risks | 剩余风险 |
| rollback | 回退方式 |

## 页面 6：提交材料与 AI 留痕页

### 目标

让参赛团队在演示结束后直接整理课题 §8.2 所需材料，并向评审说明 AI 如何被使用、审阅和约束。

### 内容分区

| 分区 | 展示内容 |
|------|----------|
| 基础信息 | 项目名称、课题名称、团队名称、成员与分工 |
| 功能说明 | 5-8 句端到端使用流程摘要 |
| 交付材料 | Demo 链接、演示视频链接、源代码仓库链接、README 状态 |
| 技术说明 | 架构图、核心技术栈、关键工程难点与解决方案 |
| AI 使用说明 | 模型清单、用途、Prompt / Skill 迭代、人工审阅点 |
| 结果说明 | ≤3 条项目亮点，**须映射 §2.2 六项** |

### AI 留痕摘要

```text
┌────────────────────────────────────────────────────────────┐
│ AI 调用摘要（代码级 P0 · rules 示例）                        │
│ clarify  rules-first-p0       tokens 0     0ms   reviewed │
│ plan/edit/verify/pr draft 由 plan.md、diff.patch、          │
│ verification.json 和 pr-draft.md 表达，不写入 ai-calls.jsonl│
└────────────────────────────────────────────────────────────┘
```

代码级 P0 可用 `rules-first-p0`。**课题完成** 须展示 **§2.2 验收口径** 的 LLM clarify：`ai-calls.jsonl` 非零 tokens，且输入为 **故意模糊/缺边界** 需求（见 [`06-plan` H4](./06-plan.md#冲刺关键路径进度-ssot)）；早期清晰 L1 输入的探索 run **不计入** #6。

### 关键状态

| 状态 | 含义 | 用户动作 |
|------|------|----------|
| incomplete | 必填材料缺失 | 补充链接或说明 |
| ready_for_submit | 必填材料齐备 | 导出提交摘要 |
| needs_review | AI 使用说明或工程难点未确认 | 人工审阅后确认 |

## 人工介入点

| 介入点 | 触发条件 | 用户可做什么 |
|--------|----------|--------------|
| 需求确认 | 澄清后 | 修改目标、范围、验收标准 |
| 方案确认 | 代码生成前 | 调整模块边界或 Skill |
| 测试失败 | lint / 单测失败 | 查看日志、修改需求、**从阶段重放**（`resume-from-stage`，待 H6–H7）或 `retry` 新 run |
| PR 前 | 验证完成后 | 编辑 PR 描述或停止提交 |

## 状态标记

| 标记 | 含义 |
|------|------|
| ⏳ | 等待中 |
| 🔄 | 执行中 |
| ✅ | 已通过 |
| ❌ | 失败 |
| 🔔 | 需要人工确认 |
| 🧾 | 已生成证据 |

## 演示脚本

**代码级 P0**（rules）演示覆盖 L1 阅读量主线；**课题完成** 视频须覆盖 [`06-plan` 冲刺关键路径](./06-plan.md#冲刺关键路径进度-ssot) 中 §2.2 六项，至少包含：

| # | 演示触点 | UI / 证据 |
|---|----------|-----------|
| 1 | 第三 Skill 只加文件 | git diff + Skill 注册表 |
| 2 | plan 后改输入，只重跑下游 | `resume-from-stage` 按钮 + 保留上游 `plan.md` |
| 3 | L2 跨栈 | plan 影响矩阵 + 跨栈 diff |
| 4 | 非零 LLM metrics | AI Usage 面板 + `ai-calls.jsonl` |
| 5 | 相似需求召回入 plan | History + plan「历史参考」段落 |
| 6 | 模糊输入 LLM 追问 | 对话区多轮澄清 + `requirement.md` |

| 等级 | 适合展示的能力 |
|------|----------------|
| L1 | §2.1 MVP：澄清到 PR 草稿（rules 或 LLM） |
| L2 | §2.2 #3：前后端+模型+测试联动（**必达**） |
| L3 | §2.2 #6：模糊/矛盾输入主动追问（**必达**） |

## 设计结论

本设计须支撑 **§2.2 六项全部必达**：阶段可暂停与 `resume-from-stage`、L2 跨栈方案页、AI 监控面板、历史召回入 plan、LLM 模糊澄清 UI。
