# 题目理解：端到端交付全栈项目的超级个体

## 一句话理解

本题不是要求做一个能聊天的 AI 页面，而是要求做一个能把 PM 的自然语言需求推进到 Conduit 真实仓库可审阅 PR 的交付系统。

## 源题关键约束

源题 PDF 的最低交付要求可以压缩为五条硬约束：

1. 至少跑通一条端到端需求链路：PM 输入 -> 澄清 -> 方案 -> 模块定位 -> 代码生成 -> 写入 Conduit 实仓 -> lint / 单测 -> PR。
2. sandbox-repo 必须基于 `TonyMckes/conduit-realworld-example-app` fork 或裁剪子集，不能使用无关 mock 仓库。
3. 系统必须同时包含前端对话页、Node 后端、AI 编排层，不能只做单端 demo。
4. 演示必须能在合理时间内完成 L1 级别需求，并展示真实 diff 与验证结果。
5. 源题 §2.1 写「使用 `doubao-seed-2.0 lite` EP/API key（见 §八）」；凭据细节在 §7.1。**忽略 §7.1 不等于 MVP 不需要任何真实模型**——见下文「模型策略」与「完成定义分层」。
6. 源题 **§2.2 评判亮点六项全部必达**（与 §2.1 MVP 并列，不是可选加分）；见下文「§2.2 评判亮点」。

## 两类「用 AI」不要混用

维护本仓库或实现仓时，常把两件事混在一起，必须分开：

| 维度 | 谁在用 | 做什么 | 豆包 | 其它 LLM（OpenAI / Anthropic 等） |
|------|--------|--------|------|-----------------------------------|
| **A. 开发协作** | 开发者 + Cursor / 其它编码 Agent | 写代码、改 `docs/` 设计文档 | **团队约定：不用豆包改文档** | 可用 |
| **B. 产品运行时** | PM 使用已交付的「超级个体」系统 | clarify 理解需求、（可选）其它 Agent 阶段 | 可选；非硬性 | **可用**；在实现仓 `docs/reports/submission/ai-usage.md` 写清模型清单与用途 |

要点：

- **「不使用豆包」** 若指 A：只约束你怎么开发、怎么改文档，**不**等于产品里不能接任何模型。
- **「理解需求」** 在 B 里由 **澄清 Agent** 完成：须满足 §2.2 **澄清深度**（真实 LLM、主动追问模糊需求，不得硬编）。
- **Conduit 业务应用**（sandbox-repo 里的博客）**不需要** LLM。

## 模型策略（产品运行时 · B）

| 层级 | 目标 | 约定 |
|------|------|------|
| **忽略 §7.1** | 凭据与配额 | 不转写示例 EP/API key、curl、TRAE 配置；凭据只走本地环境变量 |
| **代码级 P0** | 胶水链路可跑 | `AI_MODE=rules` 可跑通固定 L1 → Conduit 写入 → lint/单测 → PR 草稿；**不替代** §2.2 六项 |
| **课题完成 / 提交答辩** | 6.10 前对外提交 | §2.1 MVP + **§2.2 六项全部有证据** + §8.2 材料；LLM 须 **模糊输入验收 run**（[`06-plan` H4](./06-plan.md#冲刺关键路径进度-ssot)） |
| **是否必须用豆包** | 合规 5% | **否**；§7.2 允许多模型补充，须在材料中声明清单与用途。**团队决策（2026-05-21）**：P1-5 跳过，不另补豆包 run；§2.2 #6 以 `mimo-v2.5` 模糊验收 run 举证 |
| **禁止** | 核心链路 | mock 仓库 / mock 模型输出 / 伪测试替代 Conduit 写入与验证 |

## 对“超级个体”的定义

本题里的“超级个体”不是一个单体 Agent，而是一个由人和系统共同构成的交付单元：

| 组成 | 职责 |
|------|------|
| PM | 输入自然语言需求，回答澄清问题，确认需求卡片 |
| 人类负责人 | 审阅方案、边界、验证结果和 PR 材料 |
| Frontend Console | 展示对话、阶段、证据、diff、验证和 PR 草稿 |
| Backend API | 管理运行状态、事件流、人工确认、diff 和 PR 草稿接口 |
| Orchestrator | 串联 clarify / plan / edit / verify / pr 阶段 |
| Agents | 分别承担澄清、规划、编码、验证、PR 准备职责 |
| Skills | 承载新增字段、新增页面、新增筛选、新增交互等需求模式 |
| Sandbox Adapter | 连接 Conduit 真实仓库，负责读取、写入、diff、命令执行 |
| Evidence Store | 留存需求、方案、diff、验证、AI 调用和提交材料 |

## 核心命题

题目的关键不是“AI 会不会写代码”，而是系统能否稳定指挥 AI 完成一条可验证的工程交付链路。

因此，设计优先级为：

1. 先证明 §2.1 真实链路（L1 端到端 + Conduit 实仓 + 验证 + PR 证据）。
2. **并行补齐 §2.2 六项**——每一项须有可演示、可审计证据，不可留到「答辩再说」。
3. L1 主线跑通后，用 **L2 跨栈题** 证明跨栈一致性；用 **第二/第三个 Skill 文件** 证明抽象到位。
4. 先让失败与缺口暴露清楚，再谈体验抛光。

## §2.2 评判亮点（全部必达）

源题 §2.2 原文为「评判亮点」，本设计基线 **按六项硬性交付** 执行（与 §6.2 加分锚点一致，但不再视为可选）：

| # | 亮点 | 必达含义 | 最低证据 |
|---|------|----------|----------|
| 1 | **抽象到位** | 新增需求模式 **仅新增 1 个 Skill 文件**，不改 Orchestrator/Agent 主干 | ≥2 个 Skill 注册；现场或录屏演示第 3 模式只加 1 个 Skill 文件即可接入 |
| 2 | **断点重放** | 流程 **任一阶段** 可暂停、改输入、**只重放下游**；事件溯源持久化 | 阶段事件流 + `resume-from-stage`（或等价 API）；演示在 plan 改边界后只重跑 edit→verify→pr |
| 3 | **跨栈一致性** | 后端字段变更 **自动驱动** 前端类型/Mock/调用点/测试同步检查 | 至少 1 条 **L2 跨栈** run（如封面图/草稿）；`plan.md` 含前后端+模型+测试影响矩阵且 diff 跨栈 |
| 4 | **可观测性** | 每次 AI 调用记录 tokens/延迟/成本；**简易监控面板** | `ai-calls.jsonl` + Web **AI Usage / 成本汇总面板**（含真实 LLM 调用的非零 metrics） |
| 5 | **业务上下文反哺** | 历史需求结构化沉淀；相似新需求 **自动召回** 旧方案 | `history-recall.json` 命中并入 plan；演示相似 L1/L2 输入召回历史 Skill/文件/风险 |
| 6 | **澄清深度** | 识别 PM 模糊/冲突/缺边界需求并 **主动追问**，非硬编 | 真实 LLM clarify；**故意模糊**输入 run 的 `requirement.md` 含结构化问题与待确认项（清晰 L1 原句 run **不计入**） |

详细规格见 [03-spec.md §2.2](./03-spec.md#22-评判亮点规格)；验收见 [11-acceptance.yaml](./11-acceptance.yaml) 中 F-001、F-003、F-007–F-010、F-012。

**§2.2 当前实现状态**、冲刺任务与剩余外部 gate 见 [03-spec.md 当前实现状态](./03-spec.md#当前实现状态)、[06-plan.md §2.2 六项追踪](./06-plan.md#22-六项追踪) 与 **[06-plan 冲刺关键路径](./06-plan.md#冲刺关键路径进度-ssot)**。截至 2026-05-22，六项已有代码 / run 证据；答辩录屏和 §8.2 外部提交仍待人工。

## 完成定义分层

不要把「代码级 P0」「最终提交」混为一谈：

| 层级 | 何时可声称 | 必须满足 |
|------|------------|----------|
| **代码级 P0** | 本地/CI 胶水链路跑通 | §2.1 三端 + Conduit diff + lint/单测 + PR 草稿（可用 `AI_MODE=rules`） |
| **课题完成 / 提交答辩** | 6.10 前 | §2.1 + **§2.2 六项证据** + §8.2（公开 AI 系统主仓并内含 `sandbox-repo/`、Demo、视频等）；LLM 验收见 H4 |

## P0 成功标准（代码级）

| 标准 | 证明方式 |
|------|----------|
| 三端真实存在 | 实现仓库包含 `apps/web`、`apps/api`、`services/orchestrator` 等真实代码 |
| 真实 Conduit 写入 | sandbox-repo 指向 Conduit fork / clone，并产生 git diff |
| L1 需求闭环 | 至少一个 L1 需求从输入推进到 PR 草稿 |
| 验证结果可见 | lint / 单测命令、退出码、摘要和日志路径可查 |
| 失败可恢复 | 失败展示清楚；完整 §2.2 须 **阶段级断点重放**（不仅是 `retry` 新 run） |
| §2.2 六项 | 见上表；**课题完成前逐项闭合** |
| 提交材料 | 见「课题完成 / 提交答辩」行 |

## 提交与 PR 边界

- **§8.2 对外提交**：公开 `bytedance-implementation/` 作为 AI 系统主仓，并让仓内包含 `sandbox-repo/`。评审从这个公开仓查看系统源码、run 证据、submission 材料和 Conduit diff 来源。
- **不是必需事项**：直接向 `TonyMckes/conduit-realworld-example-app` 上游仓库提交 PR。上游 Conduit PR 只有在需要展示真实远端 PR 能力、且完成 fork/分支权限配置后才做。
- **P0 必需证据**：`pr-draft.md` 表明系统已生成可提测 PR 材料；真实 draft PR URL 属于 H17 可选远端补强，不替代 §8.2 公开主仓。

## P0 推荐演示主线

P0 主线对齐源题 §9.1 L1：**文章列表加阅读量——前端假数据即可，不改后端**。

1. PM 输入：“给文章列表加阅读量展示。”
2. 澄清 Agent 确认：展示位置、假数据来源/默认值、**明确排除后端 schema/API/数据库**。
3. Planning Agent 定位文章列表组件与样式文件（仅前端路径）。
4. Coding Agent 基于 `article-list-display-field` Skill 写入 Conduit 前端文件。
5. Verification Agent 运行真实 lint / 单测；若缺测试则显式记录缺口。
6. PR Agent 生成分支名、PR 标题、描述、验证结果和风险。
7. 提交材料页汇总 Demo、公开 AI 主仓链接、README、架构图、AI 使用说明和工程难点。

## 演示分层（§2.2 覆盖）

| 演示 | 用途 | 题型 |
|------|------|------|
| **主线 run** | §2.1 MVP + 抽象/反哺/可观测（部分） | L1 文章列表阅读量（前端假数据） |
| **跨栈 run** | §2.2 #3 跨栈一致性 | L2 如文章封面图 / 草稿（§9.2） |
| **模糊需求 run** | §2.2 #6 澄清深度 | 故意缺边界输入 + LLM 追问 |
| **Skill 扩展演示** | §2.2 #1 抽象到位 | 现场只新增 1 个 Skill 文件接入新模式 |
| **断点重放演示** | §2.2 #2 | plan 后改输入，只重放下游 |

## 本文档仓库边界

根层文档只负责题目理解、PRD、体验设计、技术规格、执行计划、架构决策、验收和追踪基线。

真实前端、后端、AI 编排、Conduit sandbox、验证报告、PR 证据和演示材料必须在 `bytedance-implementation/` 实现子树完成；§8.2 公开发布时将该子树作为 AI 系统主仓发布。
