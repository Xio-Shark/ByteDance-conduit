# 产品需求文档：Conduit 超级个体端到端交付系统 v1.0.0

## 项目定位

本项目的目标是实现一个面向 Conduit 开源全栈仓库的“超级个体”：让产品经理用自然语言提出需求后，系统能够完成需求澄清、方案拆解、模块定位、代码生成、自动化验证和 PR 提交流程。

这里的“超级个体”不是单个聊天机器人，而是一个由前端对话入口、Node 后端服务、AI 编排层、Skill 注册体系、Conduit 沙箱仓库、测试与 PR 证据共同组成的交付系统。

## 设计目标

1. 在真实 Conduit 仓库上跑通至少一条 L1 需求链路。
2. 三端齐备：前端对话页、Node 后端、AI 编排层均有真实实现。
3. 每一步都可被人类查看、修改、暂停和继续。
4. 生成结果必须写入 Conduit fork 或 sandbox-repo，并通过 lint 与单测基础验证。
5. 新需求模式应优先通过新增 Skill 接入，而不是改动主流程。
6. **§2.2 评判亮点六项全部达到**（抽象到位、断点重放、跨栈一致性、可观测性、业务上下文反哺、澄清深度）。
7. 交付材料必须能支撑现场评审：Demo、演示视频、公开仓库、README、架构图、AI 使用说明和工程难点说明。

## 目标用户

| 用户 | 关注点 |
|------|--------|
| 产品经理 | 用自然语言表达需求，看到澄清问题、方案和结果 |
| 参赛团队 | 在 3 周内完成可演示、可提交、可答辩的系统 |
| 技术评审 | 判断系统是否真实接入仓库、测试和 PR，而非 mock demo |
| 维护者 | 复用 Skill、上下文工程和编排经验到后续需求 |

## 北极星指标

> 输入一句 PM 需求后，系统能在合理时间内对 Conduit 仓库生成可审阅代码变更，完成 lint / 单测验证，并产出可提交 PR 的证据链。

## P0 功能需求

### F-001 PM 需求输入与澄清

**用户故事**：作为 PM，我希望输入一句模糊需求后，系统能主动识别缺失信息并提出必要澄清，而不是直接硬编代码。

**验收标准**：
- 支持在前端对话页输入自然语言需求。
- 澄清 Agent 能输出结构化问题、默认假设和待确认项。
- **§2.2 澄清深度**：对故意模糊/冲突/缺边界输入，真实 LLM 须主动追问，禁止未澄清直接写代码。
- 对 L1 需求可在一次澄清后形成可执行需求卡片。

### F-002 方案拆解与模块定位

**用户故事**：作为开发负责人，我希望系统先说明要改哪些模块、为什么改，再进入代码生成。

**验收标准**：
- 能读取 Conduit 仓库结构并定位相关前端、后端或测试文件。
- 能输出方案摘要、影响范围、风险和任务拆分。
- L1 需求必须定位到真实文件路径。

### F-003 Skill / Agent / Orchestrator 分层

**用户故事**：作为维护者，我希望新增需求模式时主要新增 Skill，而不是改主流程。

**验收标准**：
- 系统区分 Skill、Agent、Orchestrator 三层职责。
- 至少内置 **2 个** Skill（L1 + 扩展模式）；**§2.2 抽象到位**：接入第 3 模式时 **只新增 1 个 Skill 文件**，不改主干。
- Skill 声明输入、输出、适用场景、验证方式。

### F-004 真实 Conduit 仓库写入

**用户故事**：作为评审，我需要看到系统把代码写入真实 Conduit fork，而不是写入 mock 仓库。

**验收标准**：
- sandbox-repo 基于 `TonyMckes/conduit-realworld-example-app` fork 或裁剪子集。
- 代码变更落在真实 Conduit 文件路径。
- 变更可通过 git diff 查看。

### F-005 自动化验证闭环

**用户故事**：作为技术评审，我希望系统用自动化验证支撑结果，而不是只展示模型输出。

**验收标准**：
- 至少运行 lint 和相关单测。
- 缺失脚本或测试失败必须显式显示失败原因。
- 结果页面或报告展示命令、退出码、摘要和日志路径。

### F-006 PR 提交准备

**用户故事**：作为开发负责人，我希望系统最终生成可审阅 PR 所需材料。

**验收标准**：
- 系统能生成分支名、commit 摘要、PR 标题和 PR 描述。
- PR 描述包含需求、方案、变更文件、验证结果和风险。
- P0 可先停在“可提交 PR”状态；若接入远程权限，则可真实创建 PR。

### F-011 提交材料完整度

**用户故事**：作为参赛团队，我希望系统能沉淀最终提交所需材料，避免演示完成但答辩材料缺失。

**验收标准**：
- README 说明项目简介、依赖环境、启动步骤、目录结构、配置位置和 AI 使用边界。
- 产出系统架构图或等价架构说明，覆盖前端、后端、模型层、Skill 注册表、Orchestrator 和 sandbox-repo。
- 产出 3-8 分钟演示视频脚本或录制素材清单，覆盖核心场景、关键功能、亮点与结果。
- 产出至少 2-3 个关键工程难点与解决方案说明。

### F-012 AI 使用合规与过程留痕

**用户故事**：作为技术评审，我希望看到系统如何使用 AI、如何记录调用过程、如何避免密钥和内部数据泄漏。

**验收标准**：
- AI 调用记录包含阶段、模型、输入摘要、输出摘要、tokens、延迟、成本和结果状态。
- README 或 AI 使用说明列出使用的模型、用途、Prompt / Skill 迭代策略和人工审阅机制。
- 官方下发模型资源只通过本地环境变量配置，不把真实 API key 写入仓库。
- 如使用额外模型，必须在材料中说明模型清单、用途和合规边界。

### F-007 断点重放（§2.2 #2）

**用户故事**：作为 PM，我能在任一阶段暂停、修改输入，并只重跑后续阶段，且历史事件可追溯。

**验收标准**：
- 阶段事件持久化（SSE + 归档）；可从指定阶段 `resume-from-stage`（或等价 API）只重放下游。
- 重放保留原 run 证据；不强制重跑已成功上游阶段。
- 演示：plan 确认后修改模块边界，仅重跑 edit → verify → pr。

### F-008 跨栈一致性（§2.2 #3）

**用户故事**：作为评审，当需求涉及后端字段时，系统能同步列出并改动前端类型、Mock、调用点与测试。

**验收标准**：
- 至少完成 **1 条 L2 跨栈** run（推荐：文章封面图或草稿，见 §9.2）。
- Planning 输出前端、后端、数据模型、测试 **影响矩阵**；diff 覆盖多层真实路径。
- 验证包含跨栈相关 lint/单测（或显式缺口）。

### F-009 可观测性（§2.2 #4）

**用户故事**：作为维护者，我要在面板看到每次 AI 调用的 tokens、延迟、成本与成功率。

**验收标准**：
- 每次 LLM 调用写入 `ai-calls.jsonl`（含 tokens、延迟、成本、阶段、状态）。
- Web 控制台提供 **AI Usage / 成本汇总** 监控面板（非仅成功/失败文案）。
- 演示视频展示至少一次非零 tokens 的真实模型调用。

### F-010 业务上下文反哺（§2.2 #5）

**用户故事**：作为 PM，提交相似需求时系统能召回历史方案、Skill 与风险。

**验收标准**：
- 历史 `requirement.md` / `plan.md` 结构化归档；`GET /api/history` 与 `history-recall.json` 可检索。
- 相似新需求的 plan 阶段 **引用** 召回条目（Skill、目标文件、历史风险）。
- 演示：第二条相似 L1/L2 输入可见召回内容进入方案或 UI。

## 评分与交付要求

PDF 源题以六个维度评分，文档和实现仓库都应围绕这些维度组织证据。

| 维度 | 权重 | 本设计中的证据口径 |
|------|------|--------------------|
| 技术深度与创新性 | 30% | §2.2 抽象/澄清深度/上下文反哺 + Skill 分层 |
| 技术实现与工程完整度 | 25% | 三端、Conduit 写入、lint/单测、**断点重放**、**可观测面板** |
| 业务价值与场景契合度 | 20% | PM 对话流、主动追问、Skill 接入成本、历史召回 |
| 代码质量与文档 | 10% | README、架构图、关键模块说明、提交记录和可追溯证据 |
| 数据与合规性 | 5% | API key 不入库、数据脱敏、第三方模型用途声明 |
| 项目材料完整度 | 10% | Demo、视频、公开仓库、README、架构图、AI 使用说明、工程难点说明 |

必须规避的降档项：

- 过度依赖 prompt 硬编码，新增需求模式必须改主干。
- 无自动化测试兜底，仅凭“看起来对”判断结果。
- 模型调用、仓库写入或验证链路存在 mock / 伪实现。
- 只做对话 UI，未接入真实仓库和测试链路。

## 场景范围与练手题

- **L1**：§2.1 MVP 主线（文章列表阅读量等）。
- **L2**：**§2.2 跨栈一致性必达**——至少选 1 题完成端到端 run（推荐封面图或草稿）。
- **L3**：支撑 §2.2 澄清深度演示（模糊/矛盾输入）。

| 等级 | 示例题型 | 设计关注点 |
|------|----------|------------|
| L1 单模块 | 文章列表阅读量（**前端假数据，不改后端**）、Popular Tags 前 5 打标、About Me Tab、文章字数统计 | §2.1 MVP 主线 |
| L2 跨栈 | 文章封面图、评论点赞、文章草稿、最后编辑时间 | **§2.2 #3 必达**：前后端+模型+测试联动 |
| L3 挑战 | 模糊需求澄清、多模块拆解、反模式识别 | **§2.2 #6 澄清深度** 演示输入 |

## P2 扩展需求

- 第三条以上 Skill 与更多 L2/L3 题型。
- 集成测试与完整 Conduit 仓稳定落地。
- 多人协作和更完整的 CI/CD 提测链路。

## 非功能需求

| 类别 | 要求 |
|------|------|
| 真实性 | 不允许用 mock 仓库、mock 模型结果或伪测试替代核心链路 |
| 可演示性 | 能在现场合理时间内完成 L1 需求演示 |
| 可审计性 | 每次阶段输出、仓库 diff、验证结果均可追溯 |
| 可扩展性 | 新需求模式通过 Skill 注册接入 |
| 安全性 | API key 只通过环境变量配置，不写入仓库 |
| 可恢复性 | 阶段失败时显示失败点和重试入口，不静默跳过 |
| 材料完整性 | 最终提交材料覆盖基础信息、功能说明、交付材料、技术说明和结果说明 |

## 约束

- **§2.2 六项全部必达**；代码级 P0 可用 `AI_MODE=rules`；**课题完成**须模糊输入 LLM 验收 run + 六项证据（[`06-plan` 冲刺关键路径](./06-plan.md#冲刺关键路径进度-ssot)）。
- 公开提交须 **AI 主仓 + `sandbox-repo/` Conduit 子仓**（§8.2）；本文档仓不替代 AI 主仓。
- Conduit 仓库为真实实验田，P0 可使用裁剪子集，P1 起应对齐完整仓库。
- 系统必须包含前端、后端和 AI 编排三层真实实现。
- 本文档仓库只承载设计、规范、计划、验收和追踪基线；业务代码应在独立实现仓库中完成。
- 挑战开发期按 2026-05-20 至 2026-06-10 规划，2026-06-11 至 2026-06-19 进入答辩评审窗口；冲刺任务见 [`06-plan#冲刺关键路径`](./06-plan.md#冲刺关键路径进度-ssot)。

## 非目标

- 不追求无人值守生产发布。
- 不把生成代码行数作为成功标准。
- 不用套壳 ChatGPT 对话页替代仓库写入和验证链路。
- 不在本文档仓库内提交 Conduit fork 源码或真实 API key。

## 术语表

| 术语 | 定义 |
|------|------|
| 超级个体 | PM + 人类负责人 + Agent 系统 + Conduit 沙箱 + 验证证据组成的交付单元 |
| Conduit | RealWorld 示例博客项目，本课题指定实验田 |
| Skill | 面向某类需求模式的可注册能力单元 |
| Agent | 执行澄清、定位、编码、测试、PR 准备等职责的角色 |
| Orchestrator | 串联阶段、状态、事件和人工介入点的流程层 |
| sandbox-repo | 基于 Conduit fork 或裁剪子集的真实写入仓库 |
| L1 需求 | 单模块、低风险、可快速演示的需求 |
| 提测 | 具备 PR 材料、验证结果和可审阅 diff 的交付状态 |
