# 提交优化计划（基于比赛文档 + 实现现状核查）

> 本文以课题文档《实现一个可以端到端交付全栈项目的「超级个体」》六维权重为标尺，**并对照当前仓库实际实现**给出优化动作。
> 目标：把作品打到「优秀 ≤10」乃至「卓越 1」的竞争位，而不仅是「完赛合格」。
> 最后一次现状核查：2026-06-09，基于 `services/`、`apps/`、`scripts/` 源码实读。

## 零、实现现状核查（2026-06-09，决定后续动作优先级）

下表是基于源码实读的判定，是本计划所有优化动作的事实基础。**「真实落地」直接拿去答辩；「部分落地」需补强或诚实降级话术；「伪实现 / 仅骨架」是高危项，必须先处理。**

| 能力 | 判定 | 源码实况 |
|------|------|----------|
| 模型调用（默认 rules 链路） | **伪实现（高危）** | `AI_MODE=rules` 下 requirement/plan 由正则关键词匹配 + 硬编码返回值产出；`ai-calls.jsonl` 记 `model: "rules-first-p0"`，`tokens/latency/cost` 全为 0。默认 `npm run run:p0`、README 快速启动、关键证据表均走此链路。 |
| 模型调用（llm 链路） | 真实落地 | `AI_MODE=llm` 经 `llmClient.js` 发真实 `fetch` POST，token 取自 API `usage`，latency 实测，cost 公式估算。但非默认路径，README 降级为「可选展示」。 |
| 代码生成（编码阶段） | **模板拼接，非 LLM 生成** | 即使 llm 模式，codingAgent 也是空壳 `return skill.apply(sandbox)`；skill.apply 是硬编码字符串模板。LLM 只参与 clarify + plan 两阶段。 |
| 仓库写入 | 真实落地 | `conduitSandbox.writeText` 真 `writeFile` 落盘 sandbox-repo/，editSandbox 取真实 `gitDiff` 并断言非空。 |
| 测试执行 | 真实落地 | lint/test 经 `execFile` 真 spawn 子进程，退出码从系统取，非伪造。 |
| 上下文召回 | **仅骨架** | 所谓 embedding 是本地 FNV hash 进 256 维向量做 cosine，无任何外部 embedding 模型，本质 bag-of-bigrams 近似匹配。不是语义召回。 |
| Skill 注册 | **部分落地（踩扣分项）** | `registry.js` 硬编码数组，新增 Skill 必须改主干 import + 数组项，无法「只 git add 一个文件」。 |
| 断点重放 | **真实落地（最能打）** | 暂停/恢复/阶段跳转/confirm gate 全链路，基于文件持久化，支持 resume-from-stage / continue-after-confirm / continue-after-clarification 三路。 |
| 跨栈一致性 | 部分落地 | schemaDriver + frontendGenerators 生成链真实；但 crossStackSync 只硬编码 articleDraft 一个 case，非通用引擎。 |
| Orchestrator 状态机 | 真实落地 | 5 阶段线性状态机 + JSON 快照式 checkpoint；events 是内存数组序列化，非真正 Event Sourcing。 |

**三条最高优先级动作（按 ROI 排序）**：

1. **翻转默认链路到真实模型（规避降档红线）**：README 快速启动、`run:p0`、demo 脚本默认切 `AI_MODE=llm PLAN_MODE=llm`，rules 降级为断网应急并显式标注。详见 §二.3。
2. **创新话术要么坐实要么诚实**：召回与跨栈短期做不深，就别用「语义召回 / 通用全栈联动」表述；把**断点重放推到答辩 C 位**。详见 §三 维度 1。
3. **Skill 注册去硬编码**：改目录扫描 / 自动发现，让「新增模式只加一个文件」真正成立——这是 §四最强加分点。

**一个已识别的投入偏科**：`scripts/` 37 个文件几乎全是 submission gate / next-steps 收口工具，README 近一半篇幅讲提交流程；而材料完整度只占 10% 权重（本文 §六自评为「及格线」）。工程精力压在低权重维度，核心能力却留着伪实现与骨架——投入产出比倒挂，需要把精力回拨到 30%+25% 维度。

## 一、对齐唯一事实源：官方六维权重

所有优化必须服务于官方下发的六维权重（§6）。投入产出比按权重排序，不要在低权重维度上过度打磨。

| 维度 | 权重 | 该维度「优秀」的判定锚点 |
|------|------|--------------------------|
| 技术深度与创新性 | 30% | Skill / Agent / Orchestrator 分层原创自洽；上下文召回超越暴力切片；澄清 Agent 拆解深度 |
| 技术实现与工程完整度 | 25% | Conduit 实仓稳定产出可提测代码；Lint / 单测 / 集成测试绿；token / 时延 / 成本监控；端到端可现场演示 |
| 业务价值与场景契合度 | 20% | PM 对话流自然；能识别歧义主动追问；新增需求模式接入成本低；沉淀上下文反哺 |
| 代码质量与文档 | 10% | 系统自身通过 Lint；README / 架构图 / 注释齐全；提交记录可追溯 |
| 数据与合规性 | 5% | 豆包 EP / key 仅限本课题；不外泄；第三方模型在 README 声明 |
| 项目材料完整度 | 10% | §8.2 清单全量交付：Demo / 视频 / 仓库 / README / 架构图 / AI 使用说明等 |

**结论**：技术深度（30%）+ 工程完整度（25%）+ 业务价值（20%）= 75% 决定档位。材料完整度（10%）只是「不交付就直接扣分」的及格线，不是拉开差距的地方。

## 二、先规避扣分项（§6.1，任一触发即降档）

这四条是降档红线，必须在设计阶段就杜绝，而不是事后补救。

1. **禁止 prompt 硬编码新增模式**
   - 新增需求模式必须只新增 1 个 Skill 文件，不改主干。这是 §6.2 第一加分锚点，也是 §6.1 第一扣分项的反面。
   - 验证方式：现场出新模式题时，能当场只 `git add` 一个新 Skill 文件并跑通。
   - **⚠ 当前已踩雷（§零）**：`registry.js` 硬编码数组，新增 Skill 必须改主干 import + 数组项，无法「只 git add 一个文件」。改目录扫描 / 自动发现后才解除。

2. **禁止无测试兜底**
   - 每条生成的代码变更必须带可执行的 Lint + 单测验证，且结果是「真的跑过」而非断言通过。
   - 验证方式：run 证据里保留真实 verification 输出（命令、退出码、用例数）。
   - **现状（§零）**：通过。lint/test 经 `execFile` 真 spawn 子进程，退出码从系统取，非伪造。

3. **禁止关键能力 mock / 伪实现**
   - 三个绝不可 mock 的环节：**模型调用、仓库写入、测试执行**。
   - 主演示链路就应该真实调用模型、真实写入 Conduit、真实跑测试。把「稳定演示用的规则模式」作为兜底，而不是作为答辩主路径展示——否则评审会判定为伪实现。
   - **⚠ 当前已踩雷（§零，最高危）**：默认链路 `AI_MODE=rules` 完全不调模型，requirement/plan 由正则 + 硬编码产出，token=0。仓库写入和测试执行是真实的，但**模型调用这一环在默认路径上就是 mock**。
   - **动作**：README 快速启动、`run:p0`、demo 脚本默认切 `AI_MODE=llm PLAN_MODE=llm`；rules 仅作断网应急且显式标注「非答辩主路径」。同步准备多条真实 LM 的 L1/L2 稳定 run 作为证据。

4. **禁止套壳 ChatGPT UI**
   - 不能只做一个对话窗口贴模型输出。UI 必须可见地接入：需求卡片 → 方案 → 真实 diff → Lint/单测结果 → PR 草稿，每一步都有结构化展示。

## 三、按维度的优化意见

### 维度 1：技术深度与创新性（30%，重中之重）

文档点名了三个创新观测口径，逐一做深：

1. **Skill / Agent / Orchestrator 分层要「原创且自洽」**
   - 三层职责必须边界清晰：Skill = 需求模式知识（声明式，可注册）；Agent = 单阶段能力（澄清/计划/编码/验证/PR）；Orchestrator = 状态机 + 事件溯源 + 人工介入点。
   - 创新点应体现在「注册式扩展」而非「修改式扩展」——这是文档反复强调的核心壁垒。
   - **现状（§零）**：分层存在且可运行，但 Skill 注册是 `registry.js` 硬编码数组（修改式扩展，踩扣分项），codingAgent 是空壳 `return skill.apply(sandbox)`，requirementAgent 是 if-else 正则分支。
   - **动作**：① 把 registry 改成目录扫描 / 自动发现，让新增 Skill 真正「只加一个文件」；② 答辩时坦白 codingAgent 是「Skill 模板驱动」而非「LLM 写码」，避免被追问穿帮。

2. **上下文召回要「超越 RAG 暴力切片」**
   - 文档明确把「精准切出刚好够用的上下文」列为核心挑战之首。
   - 优化方向：基于仓库结构 / 符号 / 历史需求的语义召回，而不是整文件塞给模型。要能讲清楚「为什么这样切，省了多少 token，避免了什么幻觉」。
   - **现状（§零）**：仅骨架。所谓 embedding 是本地 FNV hash 进 256 维向量做 cosine，无外部 embedding 模型，本质 bag-of-bigrams。换说法的相似需求大概率召回失败。
   - **动作（二选一，不要中间态）**：① 坐实——接真实 embedding 模型（豆包 / 第三方），让召回经得起现场换词测试；② 诚实降级——答辩话术改为「基于 token 重叠的历史方案复用」，不碰「语义召回」字眼。**虚报创新在 30% 维度是高危。**

3. **澄清 Agent 的拆解深度**
   - 不能一问一答就结束。要能识别 PM 模糊需求中的**多个歧义维度**，分轮追问，最后结构化成可执行的需求 DSL。
   - 这是 §2.2 第 6 亮点，也是 L3 题型的核心考点（模糊需求的主动澄清）。
   - **现状（§零）**：`clarifyWithLlm.js` 在 llm 模式下是真实多轮 LLM 澄清，有 token 校验——这是真能打的点，但只在非默认链路生效。配合 §二.3 翻转默认模式后才能现场展示。

### 维度 2：技术实现与工程完整度（25%）

1. **必须在 Conduit 实仓「稳定」产出**：不是跑通一次，而是同一类需求可重复跑通。准备多条 L1/L2 链路作为稳定性证据。
2. **三种测试都要绿**：Lint + 单测 + 集成测试。文档明确列了「集成测试绿」，不要只做单测。
3. **可观测性要落到面板**：token / 时延 / 成本必须有简易监控面板（§2.2 第 4 亮点）。不只是日志，要有可视化。
4. **端到端必须现场可演示**：合理时间内完成 L1 并通过 Lint + 单测（§2.1 第 4 条）。

> **现状（§零）**：仓库写入、测试执行、状态机、断点重放均真实落地，工程底座扎实。但有两个隐患：① 默认稳定链路是 rules 伪模型（见 §二.3），「稳定产出」目前靠硬编码而非真实模型，翻转默认模式后需重新验证 LM 链路的稳定性；② 可观测性面板的 token/成本数据在 rules 模式下全为 0，只有 llm 模式才有真实数值——面板必须配合真实 LM 链路演示才有意义。
>
> **一个投入提醒**：`scripts/` 37 个文件几乎全是 submission gate / next-steps 收口工具（10% 权重），而本维度（25%）的核心能力还留着伪实现。把工程精力从提交流程回拨到「真实 LM 链路稳定性 + 可观测性面板真实数据」上，ROI 更高。

### 维度 3：业务价值与场景契合度（20%）

文档从 PM 视角给了明确口径，逐条对照：

1. **对话流自然**：PM 用自然语言说需求，不需要懂技术术语。
2. **错误反馈友好**：澄清失败、方案有风险、测试不过时，要给 PM 看得懂的反馈，而不是抛栈。
3. **操作路径符合直觉**：追加澄清、修订方案、调整模块边界、修订测试用例——这些人工介入点都要在 UI 上可达。
4. **新增模式接入成本可量化**：能说出「新增一个需求模式 = 改 N 个文件，生效时间 X」，且 N 越小越好（理想是 1）。
5. **上下文反哺可见**：相似新需求自动召回旧方案，并在 UI 上展示「我参考了哪条历史需求」。

### 维度 4：代码质量与文档（10%）

1. 系统自身代码通过 Lint（不只是生成的代码）。
2. README 要覆盖：项目简介、依赖环境、启动步骤、目录结构、配置说明、**API key 配置位置**（§8.2 明确要求）。
3. 系统架构图必须展示：前端 / 后端 / 模型层 / Skill 注册表 / Orchestrator / sandbox-repo 的调用关系（§8.2 技术说明项）。
4. 提交记录可追溯：commit 历史清晰，能看出迭代过程。

### 维度 5：数据与合规性（5%）

1. 豆包 EP / API key 仅限本课题，**绝不写入任何派生文档、源码、提交材料**——只通过本地环境变量使用。
2. 若使用第三方模型（OpenAI / Anthropic / 开源模型），必须在 README 显式声明**模型清单与用途**（§7.2 第 5 条）。
3. 涉及内部业务数据必须脱敏。

### 维度 6：项目材料完整度（10%，及格线）

按 §8.2 清单逐项交付，缺项直接扣分：

- [ ] 项目名称 / 课题（统一命名）
- [ ] 团队名称与成员名单（姓名、学校、专业、角色）
- [ ] 分工说明（可选加分）
- [ ] 端到端使用流程（5-8 句，PM 从输入到拿到可提测 PR）
- [ ] 在线 Demo 链接（本地也行，需可访问或录屏替代）
- [ ] 演示视频链接（**3-8 分钟**，展示核心场景 + 亮点 + 结果）
- [ ] 源代码仓库链接（公开，含 AI 系统主仓 + Conduit fork 子仓）
- [ ] README / 运行说明
- [ ] 系统架构图
- [ ] 关键工程难点与解决方案（**至少 2-3 个**：上下文召回精度 / 跨栈一致性 / Skill 热加载 / 断点重放等）
- [ ] 核心技术栈（前端 / 后端 / 数据库 / 中间件 / 部署 / 云资源）
- [ ] 项目亮点 / 创新点（**≤3 条**，对应 §6.2 命中的加分锚点）

## 四、命中加分锚点（§6.2，用于卓越项目排序）

这五条是评审横向排序时拉开「优秀」与「卓越」的依据。建议**全部命中并可现场演示**，每条都准备一个 30 秒的演示动作。「当前成色」列基于 §零 核查，决定哪些能直接演、哪些先补强或降级。

| 加分锚点 | 优秀程度的演示目标 | 当前成色（§零） |
|----------|---------------------|------------------|
| 1. 抽象到位 | 现场出新模式题，只新增 1 个 Skill 文件即支持，不改主干（最强加分点，务必能现场做） | ⚠ 踩雷：registry 硬编码数组，需改主干。改自动发现后才能现场演 |
| 2. 断点重放 | 流程任一阶段可暂停 → 修改 → 重放下游；事件溯源式持久化，能从中途恢复 | ✅ 真实落地，最能打。建议推到答辩 C 位（注：是 JSON 快照式 checkpoint，非严格 Event Sourcing，话术别夸大） |
| 3. 跨栈一致性 | 后端字段变更自动驱动前端类型 / Mock / 调用点更新，现场改一个字段看全栈联动 | △ 部分：生成链真实，但 crossStackSync 只硬编码 articleDraft 一个 case。只能在预设 Skill 上演 |
| 4. 可观测性 | 每次 AI 调用记录 tokens / 延迟 / 成本，有面板，能跨 run 汇总 | △ 面板存在，但 rules 模式数据全为 0。必须配 llm 链路才有真实数值 |
| 5. 业务上下文反哺 | 历史需求结构化沉淀，相似新需求自动召回旧方案并展示参考来源 | ⚠ 仅骨架：FNV hash 伪向量，非语义召回。换词测试易失败，需坐实或降级话术 |

> **答辩排序建议**：把**断点重放**作为唯一可放心主推的卓越锚点。其余四条要么先补强（抽象/召回），要么限定演示边界（跨栈/可观测性），不要在未补强前当卖点讲——评审一旦现场验证失败，比不讲更扣分。

## 五、面向 L3 答辩题型的专项准备（§9.3）

文档明确 L3 重点考三个能力，建议针对性演练：

1. **模糊需求主动澄清**：准备一个故意表述不全的需求，演示系统如何追问而非硬编。
2. **多模块协同拆解**：准备一个前端 + 后端 + 数据模型联动的需求，演示「先拆解再落地」。
3. **反模式识别**：准备一个自相矛盾 / 定义不充分的需求，演示系统如何识别风险并给出合理路径，而不是盲目生成。

> 文档原话：「不只练写代码，还要练问对问题、拆对问题、验证对结果。」答辩展示重心应放在「指挥 AI 的方法论」上，而非代码行数（§7.2 第 3 条考察重点）。

## 六、AI 使用过程留痕（§7.2 第 1 条，硬性要求）

必须作为评审材料提交：

- 关键对话记录
- Skill 定义
- Prompt 版本迭代记录
- Agent 调用日志

建议保留 AI 工具的配置、版本、Prompt 策略文件，便于答辩复盘（§7.2 第 6 条）。

## 七、演示视频脚本建议（3-8 分钟）

时间有限，按权重分配镜头：

1. （30s）开场：一句话讲清「降低 PM→研发翻译成本」的价值主张。
2. （90s）端到端主线：PM 自然语言输入 → 真实模型澄清 → 方案 → 真实 Conduit diff → Lint/单测绿 → PR 草稿。**全程用真实模型调用，不用兜底规则模式**。
3. （120s）五个加分锚点各 20-30s 快闪，重点突出「新增 1 个 Skill 文件支持新模式」和「断点重放」。
4. （60s）人工介入演示：追加澄清 / 修订方案 / 改测试用例。
5. （30s）收尾：可观测性面板（tokens / 成本）+ 一句话总结创新点。

## 八、完成定义（Definition of Done）
