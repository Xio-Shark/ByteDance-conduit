## 一、课题名称

实现一个可以端到端交付全栈项目的「超级个体」

---

## 二、课题要求（简介）

### 2.1 基础要求（MVP，所有完赛队伍最低交付）

本课题的最小功能闭环（MVP）如下：

1. **端到端跑通至少 1 条需求链路**：从 PM 自然语言输入 → 澄清 Agent → 方案 → 模块定位 → 代码生成 → 写入 Conduit 实仓 → Lint / 单测通过率 → 提交 PR
2. **真实接入 Conduit 仓库**：`sandbox-repo` 必须基于 Conduit 仓库（可 fork 后裁剪），禁止使用与 Conduit 无关的 mock 仓库；允许在 P0 阶段先在裁剪子集上跑通，P1 起在完整 Conduit 上稳定落地
3. **三端齐备**：前端对话页 + Node 后端 + AI 编排（Skill / Agent / Orchestrator）三层均有真实实现，不得仅做单端 demo
4. **可现场演示**：系统需能在合理时间内完成 L1 级别需求并通过 Lint + 单测
5. **所需资源**：使用项目组下发的 `doubao-seed-2.0 lite` EP / API key（见 §八）；如需额外模型须自付费并在 README 声明

### 2.2 评判亮点（加分项，鼓励超越预期）

评判亮点用于在 MVP 之上拉开档位。本课题鼓励以下方向（命中越多档位越高，详细对照见 §七）：

1. **抽象到位**：新增需求模式仅需新增 1 个 Skill 文件，不改主干
2. **断点重放**：流程任一阶段可暂停 / 修改 / 重放下游，事件溯源持久化
3. **跨栈一致性**：后端字段变更自动驱动前端类型 / Mock / 调用点更新
4. **可观测性**：每次 AI 调用记录 tokens / 延迟 / 成本，附简易监控面板
5. **业务上下文反哺**：历史需求结构化沉淀，相似新需求自动召回旧方案
6. **澄清深度**：能识别 PM 模糊需求中的歧义并主动追问，而非硬编生成

### 课题总体描述

基于 AI CLI 工具、Agent SDK、开源或火山方舟模型等技术，以开源全栈博客仓库 Conduit（<https://github.com/TonyMckes/conduit-realworld-example-app>，前后端单仓 monorepo）作为实验田，实现一个可以端到端交付全栈项目的「超级个体」，要求能让产品经理在平台内通过对话完成「需求澄清 → 方案拆解 → 模块定位 → 代码生成 → 自动化测试 → 提测」全链路闭环；各阶段均可人工介入（追加澄清、修订方案、调整模块边界、修订测试用例等）；同时需设计一套可扩展的 Skill / Agent / 流程编排分层，使新需求模式、新页面类型、新规范约束能以「注册」方式接入，而非改动主干；最终在该开源仓库上稳定产出可提测的代码变更，并沉淀业务上下文反哺后续需求生成，达到「显著降低 PM→研发翻译成本、提升需求吞吐」的目的。

## 三、课题背景

- **业务侧**：选用开源博客项目 Conduit 作为实验田，仓库地址 <https://github.com/TonyMckes/conduit-realworld-example-app>（React 18 + Vite + Express 4 + Sequelize + PostgreSQL，纯 JavaScript，前后端单仓 monorepo，一条 `git clone` 即可跑起来）。选型原因：
  1. Medium 风格的完整博客，具备文章、标签、评论、用户、关注、收藏等典型全栈骨架，产品形态成熟、易于理解
  2. 高频模式化功能点（新增字段 / 新增筛选 / 新增页面 / 新增交互）天然丰富，非常适合模拟 PM→研发 的真实翻译链路
  3. 前后端同仓、技术栈简单，无任何内部依赖
  4. 体量适中，既有真实工程复杂度，又不会陷入海量代码噪声
- **工程侧**：仓库为单体全栈架构（前端 React + 后端 Node.js），模块边界清晰、体量适中，既能体现真实工程的上下文复杂度，又不会陷入基础设施泥潭，便于聚焦在 AI 编排与抽象层本身
- **AI 侧**：当前 AI 在细节代码层已足够好，真正的壁垒在业务方法论、上下文工程、流程编排、Skill 抽象——即如何指挥 AI，而非 AI 能写多少行
- **训练营目标**：通过一个贴近真实工程复杂度、但技术栈受限可控的课题，考察全栈能力、工程抽象能力、AI 心智，沉淀一套可复制的 AI 需求交付范式

## 四、核心挑战

本课题需覆盖 AI 工程业务全栈链路——数据 / 上下文 → 模型 & Agent 编排 → 后端服务 → 前端交互 → 业务闭环。下表为本课题的七大挑战维度：

| 挑战维度 | 核心问题 |
|----------|----------|
| 上下文工程 | 如何在一个真实仓库里精准切出刚好够用的上下文喂给模型，避免 token 浪费与幻觉 |
| 需求澄清 | 如何设计澄清 Agent，让 PM 的模糊描述被自动结构化为可执行的需求 DSL |
| 抽象能力 | 如何把「新增字段 / 新增页面 / 新增筛选 / 新增交互」等高频业务模式抽象为可复用 Skill |
| 流程编排 | 澄清→方案→编码→测试→提测，每一步都必须可被人类打断、修订、重放 |
| 可验证性 | 生成的代码必须有自动化测试兜底，而不是「看起来对」就提测 |
| 全栈协同 | 前端组件 / 请求 / 类型与服务端接口 / 数据库 Schema 的跨栈一致性、联调自动化 |
| 工程化落地 | 在仓库原有的构建、依赖、测试、CI 约束下，保证生成的代码可被真实提交流水线接受 |

### 6.1 命题人补充：扣分项（任一触发即降档）

1. 过度依赖 prompt 硬编码——新增需求模式必须改主干，则 Skill 抽象不及格
2. 无任何自动化测试兜底，仅凭「看起来对」判断结果
3. 关键能力 mock / 伪实现（尤其是模型调用、仓库写入），未在 Conduit 实仓真实跑通
4. 套壳 ChatGPT 式 UI——只做了一个对话窗口贴模型输出，未接入仓库与测试链路

### 6.2 命题人补充：加分锚点（用于卓越项目排序）

1. **抽象到位**：现场出一道新模式题，学员只新增 1 个 Skill 文件即可支持，不改主干
2. **断点重放**：流程任一阶段可暂停、修改、重放下游，事件溯源式持久化
3. **跨栈一致性**：后端接口字段变更能自动驱动前端类型 / Mock / 调用点更新
4. **可观测性**：每次 AI 调用记录 tokens / 延迟 / 成本，有简单监控面板
5. **业务上下文反哺**：历史需求结构化沉淀，相似新需求能被自动召回旧方案

## 九、公开练手题（供日常迭代参考）

以下题目用于帮助大家熟悉 Conduit 仓库与需求模式，**不代表答辩日最终抽题范围**。

### 9.1 L1 · 入门（纯前端 / 单模块）

1. **文章列表加阅读量字段**：在首页文章卡片上增加阅读量 icon + 数字展示（前端假数据即可，不改后端）
2. **Popular Tags 侧边栏前 5 个打标**：为接口返回的前 5 个标签增加视觉标识（纯前端取前 5，不引入排序语义）
3. **个人主页新增 About Me Tab**：在 Profile 页面现有 My Articles / Favorited Articles 之外新增一个 About Me Tab，展示 `User.bio`
4. **文章详情页新增字数统计**：在文章正文下方显示「本文共 XXX 字，预计阅读 X 分钟」，前端基于 `Article.body` 计算

### 9.2 L2 · 进阶（跨栈一致性）

1. **文章加封面图字段**：Article 模型加 `coverImage` 字段，新建/编辑文章表单支持输入 URL，列表卡片和详情页展示封面图
2. **评论支持点赞（含幂等）**：Comment 增加 `likeCount`，并设计幂等点赞机制
3. **文章草稿功能**：Article 增加 `status` 枚举（draft/published），编辑器新增「保存草稿」，列表默认过滤 draft，个人主页增加 Drafts Tab
4. **文章最后编辑时间展示**：利用 `updatedAt` 在文章详情页展示「最后编辑于 X 小时前」，同时后端保证 update 会刷新该字段

### 9.3 L3 · 挑战（仅公开题型）

答辩日 L3 题目将重点考察以下能力方向：

1. **模糊需求的主动澄清能力**：面对表达不完整、边界不清晰的需求，能否主动追问而非直接硬编
2. **多模块协同拆解能力**：面对涉及前端、后端、数据模型联动的需求，能否先拆解再落地
3. **反模式识别能力**：面对自相矛盾或定义不充分的需求，能否识别风险并给出合理实现路径

建议大家在准备时，不只练「写代码」，还要练「问对问题、拆对问题、验证对结果」。
